# Full Codebase Audit — 2026-03-25

Comprehensive line-by-line audit of every file in the `pv` codebase. Four parallel review agents covered `cmd/`, `internal/commands/`, core `internal/` packages, and utility `internal/` packages respectively.

---

## Critical — App-Breaking (Services Completely Non-Functional)

### 1. `internal/container/engine.go` — Entire Docker layer is a stub

The `Engine` struct has only `NewEngine()`, `SocketPath()`, and `Close()`. There are **zero methods** for pulling images, creating containers, starting/stopping containers, removing containers, executing commands inside containers, or streaming logs. The comment on line 19 says _"The actual implementation uses github.com/docker/docker/client and connects via the Colima Docker socket"_ — this is aspirational documentation for a feature that was never implemented. Every service command in the CLI depends on this file and every one of them is broken because of it.

**Impact:** The entire `pv service:*` command family is a facade. No Docker containers are ever created, started, stopped, removed, or inspected.

### 2. `internal/commands/service/add.go:83-89,96-100` — Image pull and container creation are fake

The "Pull image" step creates a `container.Engine`, immediately discards it with `_ = engine // Pull would happen via engine.PullImage()`, and returns `"Pulled nginx:latest"` unconditionally. The "Starting..." step sets `containerID = "" // Would be set by engine.CreateAndStart()` and returns a success string like `"MySQL 8.0 running on :3306"`. The variable `containerReady` is set to `true` (line 106) despite nothing being ready. The registry is written with `ContainerID: ""`, which means `service:status` will permanently show "stopped" even though `service:add` just told the user the service is "running".

**Impact:** Users run `pv service:add mysql`, see a spinner completing with "Pulled mysql:8.0" and "MySQL 8.0 running on :3306", and believe MySQL is running. It is not. No image was pulled. No container exists.

### 3. `internal/commands/service/start.go:48-51,68-71` — Container start is fake

Both the "start all" and "start one" code paths contain:
```go
// Docker SDK: find existing container, start it.
return fmt.Sprintf("%s started", key), nil
```
No container is looked up, no Docker API call is made. The function succeeds unconditionally. Additionally, `start.go` never updates `instance.ContainerID` in the registry and never calls `reg.Save()`, so even if the Docker call were present, `service:list` and `service:status` (which check `ContainerID != ""`) would still report the service as not running.

**Impact:** `pv service:start postgres` prints `"postgres started"` with a checkmark. Nothing happened. Status remains "stopped".

### 4. `internal/commands/service/stop.go:34-37,51-54` — Container stop is fake

```go
// Docker SDK: stop container.
return fmt.Sprintf("%s stopped", key), nil
```
No container is stopped. The spinner shows success. The `.env` fallback hooks (`applyFallbacksToLinkedProjects`) do run and modify project `.env` files — so the side effects fire but the primary action (stopping the container) does not.

**Impact:** `pv service:stop` modifies `.env` files as if the service stopped, but the container (if one existed) keeps running.

### 5. `internal/commands/service/remove.go:31-34` — Container remove is fake

```go
// Docker SDK: stop + remove container.
return fmt.Sprintf("%s removed", key), nil
```
The registry entry is deleted and Caddy configs are regenerated, so the service disappears from `service:list`. But the actual Docker container (if one ever existed) is left dangling with no way to manage it through pv.

**Impact:** `pv service:remove mysql` removes MySQL from the registry but leaves the Docker container running and orphaned.

### 6. `internal/commands/service/destroy.go:42-43,68` — Container destroy is fake + silent error swallow

The container stop/remove comment stub is the same as remove.go. The data directory deletion (`os.RemoveAll`) does work (line 47). But additionally, line 68 silently discards the Caddy config regeneration error with `_ = caddy.GenerateServiceSiteConfigs(reg)`, while `remove.go` and `add.go` both surface this error to the user via `ui.Subtle`. Inconsistent and means Caddy config failures are invisible after `service:destroy`.

**Impact:** Data is deleted but the container is not touched. Caddy errors are swallowed.

### 7. `internal/commands/service/logs.go:33-36` — Log streaming is fake

```go
// Docker SDK: ContainerLogs with Follow=true
// This would stream logs to stdout.
ui.Subtle(fmt.Sprintf("Tailing logs for %s (container: %s)...", key, instance.ContainerID))
return nil
```
The function prints one line of text and exits. No logs are ever streamed. This is the most user-visible stub because the command appears to start doing something (printing the container ID) and then immediately returns to the shell prompt.

**Impact:** `pv service:logs postgres` prints one line and exits. No logs visible.

### 8. `internal/commands/service/status.go:45-48` — Status always shows "stopped"

```go
status := "stopped"
if instance.ContainerID != "" {
    status = "running"
}
```
Because `ContainerID` is always `""` (the stub in `add.go` never assigns a real value, and `start.go` never updates the registry), every service will always display as "stopped" regardless of its actual state.

**Impact:** `pv service:status postgres` always shows "stopped" even after `service:add` reports "running".

### 9. `internal/commands/service/list.go:42-44` — List status always shows "added"

Same root cause as #8. The list command checks `svc.ContainerID != ""` to show "running" vs "added". Since `ContainerID` is always empty, all services show as "added" and never as "running".

**Impact:** `pv service:list` always shows every service as "added", never "running".

### 10. `internal/services/mysql.go:83-89` — CreateDatabase is a stub

```go
func (m *MySQL) CreateDatabase(engine *container.Engine, containerID, dbName string) error {
    _ = engine
    _ = containerID
    _ = dbName
    return nil
}
```
The comment says _"Implemented via Docker exec in the container engine layer"_ but the engine has no `Exec` method. All three arguments are discarded. Always returns nil (success) without creating any database. Called from the `CreateDatabaseStep` in the automation pipeline during `pv link`.

**Impact:** `pv link myapp` with MySQL as a registered service will report "Database created" but no SQL was executed. The database does not exist.

### 11. `internal/services/postgres.go:79-83` — CreateDatabase is a stub

Identical pattern to MySQL. All arguments discarded, always returns nil. No database is ever created in PostgreSQL.

**Impact:** Same as #10 but for PostgreSQL.

### 18. `internal/colima/recovery.go:11` — ServicesToRecover is dead code (container recovery broken)

`ServicesToRecover()` is exported and documented with a comment saying _"called on daemon start after Colima is verified running"_, but no caller in the entire codebase invokes it. The daemon startup path (`server.Start`) does not attempt to recover service containers after starting Colima. Even if it were wired up, there is no `container.Engine.Start()` method to actually restart containers — this is part of the same missing Docker layer as #1-#11. The function should be deleted as dead code and reimplemented properly when the container engine is built.

**Impact:** After a Colima VM restart (e.g., after macOS reboot), registered services are not automatically restarted. Users must manually run `pv service:start` (which itself is a stub that does nothing — see #3).

## High — Bugs That Will Bite Users

### 12. `internal/laravel/steps.go:78-80,137-139,304-306` — Wrong PHP binary path construction

Three automation steps (`GenerateKeyStep`, `InstallOctaneStep`, `RunMigrationsStep`) construct the PHP binary path as:
```go
phpBin := "php"
if ctx.PHPVersion != "" {
    phpBin = "php" + ctx.PHPVersion
}
```
This produces strings like `"php8.3"` — a bare command name that doesn't exist anywhere in the system. The pv shim is at `~/.pv/bin/php` (and delegates version resolution internally), and the actual binary is at `~/.pv/php/8.3/php`. When `exec.Command("php8.3", "artisan", ...)` is called, it fails with "command not found". This breaks `artisan key:generate`, `composer require laravel/octane`, and `artisan migrate` for any project using a non-global PHP version.

**Impact:** `pv link` automation pipeline fails for projects with a per-project PHP version. Artisan commands produce "command not found" errors.

### 13. `internal/setup/resolver.go:196-201` — Nil pointer panic if FrankenPHP fails to start

```go
srv := exec.Command(frankenphp, "run", "--config", caddyfile, "--adapter", "caddyfile")
// ...
done := make(chan error, 1)
go func() { done <- srv.Wait() }()
defer func() {
    srv.Process.Kill()  // panics if srv.Start() failed — Process is nil
    <-done
}()
```
The `defer` is registered before `srv.Start()` is called (line 191). If `Start()` fails (e.g., frankenphp binary is missing, permissions error, port in use), `srv.Process` is nil. The deferred function fires during stack unwinding and calls `nil.Kill()`, causing a nil pointer dereference panic. The `go func() { done <- srv.Wait() }()` goroutine also calls `Wait()` on a process that was never started, which returns immediately with an error that gets sent to the `done` channel but is never read (the defer panics before reaching `<-done`).

**Impact:** If FrankenPHP fails to start during `pv install` CA trust setup, the entire CLI panics instead of returning a clean error.

### 14. `cmd/update.go:40-50` — syscall.Exec failure silently returns success, skips all updates

```go
reexeced, err := selfUpdate(client)
if err != nil {
    if !errors.Is(err, ui.ErrAlreadyPrinted) {
        ui.Fail(fmt.Sprintf("pv self-update failed: %v", err))
    }
}
if reexeced {
    return nil  // returns success even though Exec failed
}
```
`selfUpdate` returns `(true, <error>)` when `syscall.Exec` fails (line 145: `return true, syscall.Exec(...)`). In the caller, the error is printed but the function returns `nil` (success) and skips all subsequent tool updates (PHP, Mago, Composer, Colima, packages). The user sees the self-update error but then the command exits cleanly with no further updates applied.

**Impact:** `pv update` fails to update any tools if the self-update binary re-exec fails. The error is printed but the exit code is 0 (success).

### 15. `internal/phpenv/resolve.go:150` — Tilde constraint `~X.Y.Z` treated as `~X.Y`

The regex `reCaretTilde` correctly matches `~8.4.0`, but the implementation discards the patch component and computes `nextMajor = "9.0"`, giving `>=8.4 <9.0`. In Composer semantics, `~8.4.0` means `>=8.4.0 <8.5.0` — the tilde at the patch level keeps the minor version fixed. Only `~8.4` (without patch) means `>=8.4 <9.0`. The test at `resolve_test.go:22` asserts the wrong behavior (`"~8.4.0"` returns `"8.5"` meaning `<9.0`), encoding the bug into the test suite and masking it from detection.

**Impact:** Any project with `"require": {"php": "~8.4.0"}` in `composer.json` could unexpectedly get PHP 8.5 or higher instead of the highest 8.4.x version.

### 16. `cmd/setup_tui.go:150` — Off-by-one: last automation item unreachable

```go
case "down", "j":
    if m.settingsCursor < len(automationItems)+1 {
        m.settingsCursor++
    }
```
`automationItems` has 10 entries (indices 0-9). The settings tab cursor maps: 0=TLD, 1=daemon, 2-11=automationItems[0-9]. The maximum valid cursor is 11. The guard `< len(automationItems)+1` evaluates to `< 11`, capping the cursor at 10. This makes `automationItems[9]` ("Reset .env on service stop") permanently unreachable — the user can never navigate to it or toggle it in the setup wizard.

**Impact:** The "Reset .env on service stop" automation setting cannot be configured via `pv setup`. Users must manually edit `pv.yml` to change it.

### 17. `internal/registry/registry.go:88-95,97-103` — Find/FindByPath return pointer to loop-local copy

```go
func (r *Registry) Find(name string) *Project {
    for _, p := range r.Projects {
        if p.Name == name {
            return &p  // p is a loop-local copy, not a reference to the slice element
        }
    }
    return nil
}
```
Go's `for _, p := range` creates a copy of each element. The returned `*Project` points to a stack-local copy that will be garbage collected. Any caller that mutates the returned pointer (e.g., `reg.Find("myapp").PHP = "8.3"`) will modify a throwaway copy and the change is silently lost. Current callers in `server/process.go` work around this by doing a separate index-based scan, but the API is a trap for future code. `FindByPath` has the identical issue.

**Impact:** Mutations via `reg.Find()` return values are silently discarded. Current callers happen to avoid this, but the API is fragile.

## Medium — Functional Gaps, Dead Code, Resource Leaks

### 19. ~~`internal/packages/updater.go` — StartBackgroundUpdater never called~~ FALSE POSITIVE

Verified: `StartBackgroundUpdater` IS called from `internal/server/process.go:128` inside the daemon supervisor. Not dead code.

### 20. `cmd/log.go:105` — File descriptor leak in log follow mode

```go
// Inside an infinite for { ... } loop, on log rotation:
f.Close()
f, err = os.Open(logPath)
if err != nil {
    return ...
}
defer f.Close()   // accumulates on every rotation, never fires
scanner = bufio.NewScanner(f)
```
The `defer f.Close()` inside the infinite polling loop accumulates a new deferred close on every log rotation event. Since the loop is intentionally infinite (exited only by process termination), these deferred calls never execute. The old file handle is correctly closed explicitly, but the newly opened file handle's `defer` never runs. Each log rotation leaks one file descriptor.

**Impact:** Long-running `pv log -f` sessions that experience log rotation will slowly leak file descriptors. On macOS the default ulimit is 256, so this would eventually cause "too many open files" errors after ~256 rotations.

### 21. `cmd/env.go:18-29` — Paths hardcoded instead of using config functions

```go
binDir := filepath.Join(home, ".pv", "bin")
composerBinDir := filepath.Join(home, ".pv", "composer", "vendor", "bin")
composerHome := filepath.Join(home, ".pv", "composer")
composerCacheDir := filepath.Join(home, ".pv", "composer", "cache")
```
The `config` package provides `config.BinDir()`, `config.ComposerBinDir()`, `config.ComposerDir()`, and `config.ComposerCacheDir()` which produce identical paths. `env.go` does not import `config` and manually recomputes these. If path conventions change in `config/paths.go`, `pv env` output silently diverges and the user's shell exports break Composer isolation.

**Impact:** Path changes in `config/paths.go` won't propagate to `pv env`, breaking the user's shell setup.

### 22. `cmd/uninstall.go:297` — "Completely uninstalled" printed regardless of failures

Multiple `ui.Step` calls can fail during uninstall (daemon stop, plist removal, DNS resolver removal, CA cert removal, `~/.pv` removal, binary removal). Each failure is displayed inline via `ui.Step`'s built-in error handling, but the final `ui.Success("pv has been completely uninstalled...")` is printed unconditionally. If every step failed, the user still sees "completely uninstalled".

**Impact:** Users may believe uninstall succeeded when critical artifacts remain on disk.

### 23. `internal/commands/service/destroy.go:68` — Caddy config error silently swallowed

```go
_ = caddy.GenerateServiceSiteConfigs(reg)
```
`remove.go` and `add.go` both surface Caddy regeneration errors to the user via `ui.Subtle(fmt.Sprintf("Could not regenerate..."))`. `destroy.go` discards the error entirely with `_ =`. Inconsistent error handling across the service commands.

**Impact:** Caddy config regeneration failures after `service:destroy` are invisible to the user.

### 24. `internal/binaries/manager.go:14-30, install.go:41-116` — Dead FrankenPHP/PHP install code in wrong directory

`binaries.FrankenPHP` and `binaries.PHP` are declared as managed `Binary` values, and `InstallBinaryProgress` has full dispatch cases for them. However, there are zero production call sites — FrankenPHP and PHP are exclusively managed by `phpenv`. Additionally, `installFrankenPHP` installs to `config.BinDir()` (`~/.pv/bin/`) which is the PATH-visible directory for shims/symlinks only. Per CLAUDE.md binary storage rules, real binaries must go in `config.InternalBinDir()` or `config.PhpVersionDir()`. If this dead code were ever activated, it would place binaries in the wrong location.

**Impact:** Dead code. No current impact, but a trap if anyone tries to use it.

### 25. `internal/packages/` — No Uninstall/Remove function exists

The `packages` package provides `Install()` and `Update()` but has no `Uninstall()` or `Remove()` function. For `MethodPHAR` packages, the PHAR file and symlink in `~/.pv/bin/` are never cleaned up. There is no command or function path to remove installed packages.

**Impact:** Installed PHAR packages (if any are added) cannot be uninstalled through pv.

### 26. `internal/automation/steps/generate_tls_cert.go:27-29` — Error messages reference Vite for all project types

```go
return "", fmt.Errorf("Vite TLS setup skipped: %w", err)
return "", fmt.Errorf("Vite TLS cert not generated (HTTPS HMR may need manual config): %w", err)
```
The step is named `GenerateTLSCertStep` and runs for all project types (`ShouldRun` returns `true` unconditionally). But error messages mention "Vite TLS" and "HTTPS HMR" which only apply to JavaScript frontend tooling. Users of non-Vite projects (vanilla PHP, Livewire, Inertia with SSR) will see confusing error messages about Vite when TLS cert generation fails.

**Impact:** Confusing error messages for non-Vite projects during `pv link`.

## Architecture Summary

The codebase has two fully working layers and one completely missing layer:

**Working:**
- PHP management (download, install, version switching, shims, multi-version support)
- DNS server (miekg/dns, AAAA support, resolver file management)
- FrankenPHP/Caddy integration (config generation, admin API, TLS certs, multi-version proxying)
- Project linking (detection, automation pipeline, .env management, Caddy site configs)
- Daemon management (launchd plist, bootstrap/bootout, PID files)
- Setup wizard (TUI, PHP/tool selection, automation settings)
- Doctor diagnostics (DNS, ports, CA trust, project health)
- Self-update mechanism
- Binary management (download, checksum verification for Composer, exposure/shims)

**Working (specs only):**
- Service definitions (port schemes, env vars, Docker image specs, Caddy routes, CreateOpts structs) — the specifications are real and correct
- Service registry (data model, persistence, project-service binding) — stores data correctly

**Completely Missing:**
- Docker execution layer (`container.Engine`) — the bridge between service definitions and actual containers
- Container lifecycle (pull, create, start, stop, remove, exec, logs, inspect)
- Database creation (MySQL/PostgreSQL `CreateDatabase` implementations)
- Container recovery after VM restart (`ServicesToRecover` is dead)
